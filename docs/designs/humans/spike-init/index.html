这是什么

spikekit 的第三个 skill：/spike-init。对一个新项目或已有项目，调研代码库、实际跑一遍「怎么搭环境 / 怎么跑测试 / lint / 类型检查」这些命令，然后产出一对规范的 agent 项目上下文文件 —— AGENTS.md（跨工具通用）+ 一个 @AGENTS.md import 它的 CLAUDE.md（Claude Code 用）。已有这些文件就完善而非覆盖。要解的痛点：agent 进项目最先卡的「环境怎么搭、测试怎么跑」—— 而模型猜的命令经常是错的。

怎么工作的

跟 Claude Code 自带的 /init 的关键区别：/init 是「分析完写下推断的命令」—— 推断可能错。/spike-init 把要推荐的命令先跑一遍，只写跑通的，跑通的打 ✅ verified <日期> 标记。下面这个真实例子（spike 里跑过的 pallets/click）最能说明问题：

✗ 只「分析后写下」（会翻车）
看到 pyproject.toml → 推断： 
pip install -e .
pytest
用系统 Python（3.9）跑 → ERROR: Package 'click' requires a different Python: 3.9.13 not in '>=3.10'，pytest 直接 8 errors。写进文档的是一套错命令。
✓ spike-init：先跑、再写
看到 uv.lock + CI 用 setup-uv → 试： 
uv sync --locked      # 自动建 .venv，挑 Python 3.12
uv run pytest         # 1493 passed, 24 skipped
亲眼跑通 → 才写进 AGENTS.md，标 ✅ verified 2026-05-13。

一次 /spike-init 大致是这么走的：

  1. 看有没有现成的 —— 有 AGENTS.md/CLAUDE.md/.cursorrules 就进「完善」模式（里面每句话当成待验证的假设），没有就「生成」模式。
  2. 调研（只读） —— 优先看 lockfile 定包管理器（uv.lock→uv、pnpm-lock.yaml→pnpm、package-lock.json→npm…），再看 CI 配置里维护者实际跑的命令、package.json scripts / Makefile / tox、.devcontainer/、README/CONTRIBUTING。不从「看到 manifest」直接猜 install 命令。
  3. 验证（真的跑） —— 跑 install（npm ci / uv sync --locked / …），跑测试，力所能及也跑 lint/类型检查。失败就退而求其次试别的、并把死路记下来。需要 DB/secret 才能跑的，如实写成前置条件，不假装跑绿。默认就跑，跑前告诉你一声，但没有关掉它的开关 —— 这是这个 skill 的核心。
  4. 完善模式：逐条对账 —— 被验证驳倒的就改（附原因）；确认的留着；验不了但不矛盾的（"release 流程在 wiki"）原样保留；引用了不存在的东西（一个根本没有的 scripts/build.sh）不静默删、问你。保留原文件结构，少改动让 diff 好审。
  5. 只问空白 —— 该有的基本都有了，只问验不了的、哪儿都没写的约定、和矛盾点，每项预填最佳猜测。
  6. 写这一对文件 + 给你看了再落盘 —— AGENTS.md（分节、紧凑、✅ verified 标记、顶部加条记录「何时由谁验证」的 HTML 注释）+ 仓库根的 CLAUDE.md（第一行 @AGENTS.md；已有就把 import 折进去不覆盖）。生成模式给预览，完善模式给 diff。

结果：一个全新的 agent 读完 CLAUDE.md/AGENTS.md，第一次就能把环境搭起来、把测试跑起来，不用瞎试。

决策 & 为什么

  • 产物是一对：AGENTS.md + @AGENTS.md 的 CLAUDE.md —— 因为 Claude Code 不直接读 AGENTS.md，只读 CLAUDE.md；而 AGENTS.md 是 Cursor/Codex/Aider 等都认的跨工具约定。只生成 AGENTS.md 的话 Claude Code 自己反而看不到。
  • 跟 /init 的本质差异 = 命令实跑验证过 —— 否则就是重复 /init。验证过的打 ✅ verified，让下游 agent 和 reviewer 一眼知道可信。
  • 写进项目级、提交进版本库、团队共享的文件（仓库根的 AGENTS.md/CLAUDE.md），不是用户机器上的 ~/.claude/ —— 「各开发阶段的固定上下文」是项目级的东西。
  • 完善模式不覆盖 —— 验不了的人类知识（release 流程、业务规则、"问某人"）有价值，保留；只对「被事实驳倒」和「引用不存在的东西」动手，后者还先问。
  • verify 默认就跑、不问、无逃生口 —— 这是 skill 的核心，不能默认关掉；跑前告诉用户一声即可。
  • skill 名 /spike-init —— 跟 spikekit 家族 spike/spike-wrap 一致，也呼应 Claude Code 的 /init；verify 步骤实跑命令，算是「用真实实验确认事实」的 spike 精神。
  • v1 只做单仓库 —— monorepo（per-package AGENTS.md、就近优先级）、生成 .claude/rules/ 或 settings.json/hooks、自动保鲜 —— 都推迟。

验证了什么

  • tj/commander.js（Node/jest）：从 package-lock.json → npm、engines.node>=20、CI 配置定出命令；实跑 npm ci（13s）→ npm test → 1367 passed。「搭环境=npm ci、跑测试=npm test、全检=npm run check、自动修=npm run fix」全部实跑确认。
  • pallets/click（Python/uv+tox）：naive 路子（pip install -e . + pytest）用系统 Python 3.9 → 翻车（要 ≥3.10）—— 完美演示痛点。对的路子：uv.lock → uv；实跑 uv sync --locked（21s，自动挑 Python 3.12）→ uv run pytest → 1493 passed / 24 skipped。
  • 「完善不覆盖」：往 commander 塞一份故意写错的 AGENTS.md（"Node 14+"、"yarn"、引用不存在的 scripts/build.sh、一句不可验证的 "release 在 wiki"），走对账逻辑 → 矛盾的改掉并附原因、缺的补全、引用不存在的提出来问、不可验证不矛盾的原样保留、保留原结构。逻辑走通了。
  • 产出形态、流程的完整提案都写好了（见设计文档的 assets/ 里的样例文件）。结论：可以实现（写 skills/spike-init/SKILL.md）。

遗留问题 & 风险

  • monorepo / workspaces —— per-package AGENTS.md、就近优先级、verify 要不要逐包跑 —— 推迟到 v1 之后。
  • 测试需要外部依赖的项目（DB/Redis/secret）—— skill 必须能识别「拿不到干净的 run」并把前置条件写成已知要求，而不是装作跑绿了。SKILL.md 里要说清，否则模型会倾向于假装。
  • verify 改动工作树（node_modules/、.venv/、可能的 lockfile 漂移）—— 如果这些没在 .gitignore 里，应警告用户。
  • 保鲜 —— provenance 注释 + ✅ verified <日期> 只是起点，re-run 是手动的；「检测 AGENTS.md 过时」的 hook 是可能的后续，但 hooks 这轮不做。
  • 跟 /init 的重叠 —— 把 /spike-init 定位成「会实跑验证的那个」，并在 description 里写明，免得搞混何时用哪个。
